0. SERIALIZATION — what the .keras file stores  (audit: model portability)
--------------------------------------------------------------------------------
   final_emotion_model.keras is a ZIP archive holding three things, so the model is
   fully self-contained and portable:
     * config.json  — the architecture (layer graph, shapes, hyperparameters)
     * model.weights.h5 — the learned weight tensors
     * metadata.json / optimizer state — so training could resume exactly
   tensorflow.keras.models.load_model() reads config.json to REBUILD the identical
   graph, then loads the weights into it — no source code needed to run inference.
   (The file is tracked with Git LFS: *.keras is large binary, so the repo stores a
   text pointer whose sha256 oid IS the model's content hash — see the manifest.)


1. INPUT
--------------------------------------------------------------------------------
   (48, 48, 1) float32, rescaled to [0, 1] (preprocessing.normalization = rescale).
   Grayscale => 1 channel. The channel axis is explicit because Conv2D needs
   (H, W, C). This is the exact tensor the pipeline emits (data.md §5, notebook 08).


2. DESIGN PHILOSOPHY — VGG-style
--------------------------------------------------------------------------------
   Stack small 3x3 convolutions and double the channels as spatial resolution halves:
   * Two stacked 3x3 convs match one 5x5's receptive field with fewer params
     (2*9=18 vs 25 per channel-pair) AND add a non-linearity => more expressive/param.
   * Doubling channels (32 -> 64 -> 128) as we pool keeps per-layer compute balanced
     and lets deeper layers build abstract features (edges -> parts -> expressions).
   * Overfitting is THE risk: every block ends BatchNorm + Dropout; augmentation and
     early stopping do the rest.


3. THE BLOCK PATTERN
--------------------------------------------------------------------------------
       [ Conv3x3 -> BatchNorm -> ReLU ] x 2   (feature extraction)
         -> MaxPool 2x2                        (downsample, small-shift invariance)
         -> Dropout                            (regularization; blocks use dropout/2)
   Repeated num_conv_blocks (=3) times, channels filters_start*2^b: 32 -> 64 -> 128.
   Head:  Flatten -> Dense(256) -> BatchNorm -> ReLU -> Dropout -> Dense(7, softmax)

5. LAYER-TYPE RATIONALE (why each piece)
--------------------------------------------------------------------------------
   Conv2D 3x3   Parameter-shared, local filters; early layers learn edges, deeper
                layers combine them into facial parts. 3x3 is VGG's core choice.
   MaxPool 2x2  Keeps the strongest activation per 2x2 window: small-shift invariance
                + cheaper compute. MAX over AVG because expression cues are sharp.
   ReLU         Non-saturating, cheap, sparse activations — the conv-net default.
   BatchNorm    Standardizes activations per batch => higher LR, steadier training,
                mild regularization. Ordered Conv -> BN -> ReLU.
   Dropout      Randomly zeros activations so no unit relies on a partner => redundant,
                generalizing features. Our primary in-network overfitting defense.
   Softmax      7 logits -> probability distribution, paired with categorical CE loss.


6. ITERATION HISTORY  (the ablation log — what was tried, what moved the metric)
--------------------------------------------------------------------------------
   Recorded newest-last. Each row is a config toggle and its observed effect. This is
   the audit's "prior iterations / why this architecture" answer.

   (a) BASELINE — simple_cnn (2 conv layers, no BN, one dense head).
       Trains fast but underfits; the in-project floor the deeper nets must beat
       (the MNIST logreg-vs-CNN habit applied to architecture). => moved to VGG-style.

   (b) ARCHITECTURE — simple_cnn -> vgg_small (stacked 3x3 blocks + BN + Dropout).
       Clear val-accuracy gain; best accuracy/complexity trade-off at 48x48. Kept as
       the default. (resnet_mini / transfer_vgg16 stayed as config options, not chosen:
       ResNet's depth was unnecessary here, and ImageNet transfer mismatches 48x48 gray.)

   (c) NORMALIZATION — tried none | rescale | standardize | CLAHE.
       rescale ([0,1]) gave stable training with no downside; kept as default.

   (d) CLEANING — deduplication removed 803 exact-duplicate images (cross-split leakage),
       + 1 constant/black frame. Class imbalance was ~16.5x (Disgust rare); enabling
       class_weight lifted minority-class recall (macro-F1) without hurting accuracy.

   (e) AUGMENTATION — basic (h-flip + rotation 10 + zoom 0.1 + shift 0.1).
       Narrowed the train/val gap (regularization); kept on. "heavy" was available but
       basic was enough at this dataset size.

   (f) HEAD — Flatten vs GlobalAveragePooling2D. Flatten kept (VGG-faithful, more
       capacity); the BN+Dropout on the head control its overfitting. GAP remains the
       first lever if the train/val gap widens (head would shrink ~1.18M -> ~33k).

   (g) CALLBACKS — EarlyStopping(restore_best_weights) + ReduceLROnPlateau(factor 0.5).
       LR dropped 1e-3 -> 5e-4 -> 2.5e-4 -> ... as val_loss plateaued; the best weights
       came from epoch 55, and training stopped at 65 (patience exhausted) — the model
       is the epoch-55 checkpoint, NOT the last epoch.

   (h) HYPERPARAMETER SEARCH (#43/#44; results/tuning/tuning_results.json).
       Random search over {learning_rate, batch_size, dropout_rate, filters_start,
       optimizer}. WINNER (rank 1): learning_rate 0.001, batch_size 64, dropout_rate 0.3,
       filters_start 32, optimizer adam. Notably the search preferred LIGHTER dropout
       (0.3 vs the 0.5 default) — with BN + augmentation already regularizing, heavier
       dropout was slowing convergence more than it helped. These are the hyperparameters
       the final model was trained with (stages.tuning: true applied them before the run).
